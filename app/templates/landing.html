Frequently Asked Questions

What is API contract drift?

API contract drift occurs when an API specification changes in ways that break client expectations. This happens when endpoints are removed, required parameters are added, or response structures change unexpectedly. SpecDrift detects these drifts automatically.

What's the difference between breaking and non-breaking changes?

Breaking changes force clients to update their code immediately (e.g., removing an endpoint, adding a required field). Potentially breaking changes might cause issues depending on implementation (e.g., changing response status codes). Non-breaking changes are safe‚Äîlike adding optional fields or new endpoints.

How do I use SpecDrift?

Upload or paste two Swagger / OpenAPI specifications, click "Compare Specs," and SpecDrift instantly shows all changes categorized by risk level.

What file formats does SpecDrift support?

SpecDrift accepts Swagger 2.0 and OpenAPI 3.0, and compatible formats in JSON or YAML. You can paste specs directly or upload files. You can even compare a Swagger 2.0 specification to an OpenAPI 3.0 specification.

Is my data private and secure?

Yes. SpecDrift processes comparisons on-the-fly without storing your specifications. No signup, no tracking. Your data stays with you.

Why should I use SpecDrift instead of manual review?

Manual review is error-prone and slow, especially with large or complex APIs. SpecDrift automatically flags issues in seconds, catches edge cases

Can I use SpecDrift in my CI/CD pipeline? In Development

SpecDrift's API endpoint accepts spec comparisons programmatically, making it ideal for automated API validation during deployments.

What happens when endpoints or fields are removed?

SpecDrift flags removed endpoints and required fields as breaking changes. Clients depending on these will fail unless they update.
